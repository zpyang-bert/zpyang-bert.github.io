## 执行摘要

MLSE（Maximum Likelihood Sequence Estimation，最大似然序列估计）是理论上最优的均衡方案，基于Viterbi算法在整个接收序列中搜索最可能的发送序列。相比DFE的逐符号判决，MLSE考虑符号间的全局依赖关系，在AWGN信道下可达理论最优性能。然而MLSE的硬件实现复杂度随时序状态数指数增长，是制约其大规模商用的核心瓶颈。本报告从算法原理、硬件架构、功耗与性能权衡、224G SerDes场景的适配与演进四个维度进行系统性深度论述。

## 第一章 MLSE算法原理与理论框架

### 1.1 问题定位：为什么需要序列估计

在高速SerDes和光纤通信系统中，信道色散导致脉冲展宽，形成码间串扰（ISI）。FFE和DFE均衡器基于逐符号判决思路，无法利用符号间的约束关系——特别是当前符号的判决错误会通过反馈路径污染后续符号（DFE的错误传播）。

MLSE的根本创新在于：将接收序列与所有可能的发送序列进行全局匹配，在整个观察窗口内寻找最可能的发送序列，而非逐个符号独立判决。这在理论上等价于在网格图（trellis）上搜索最大似然路径。

### 1.2 网格图状态建模

若信道脉冲响应长度为L+1个抽头（二进制调制），MLSE的状态数等于2^(L+1)：

| MLSE类型 | ISI影响范围 | 状态数 | 适用场景 |
|---------|-----------|-------|---------|
| MLSE-1 | 前后各1个符号 | 4 | 短距离（<100km光纤/低损耗背板） |
| MLSE-2 | 前后各2个符号 | 8 | 中距离（100-200km光纤） |
| MLSE-3 | 前后各3个符号 | 16 | 长距离（200-400km光纤） |

状态数越多，可补偿的ISI越长，但计算复杂度也越高——每增加一个状态数，复杂度翻倍。这是因为每个状态需要计算所有可能的分支转移，并执行Add-Compare-Select（ACS）操作。

### 1.3 Viterbi算法判决过程

Viterbi算法的核心步骤：

1. **分支度量计算（Branch Metric）**：对每个状态，计算接收符号与所有可能发送序列之间的度量差
   - AWGN信道下：BM = |y[k] - μ_k|² （μ_k为状态均值）
   - 状态均值由当前符号及其ISI影响范围内的符号共同决定

2. **累计路径度量更新（Path Metric）**：每状态将所有进入分支的度量与该分支的分支度量相加，得到新的路径度量

3. **幸存路径选择（Survivor Selection）**：对到达同一状态的多条路径，保留度量最小（最优）的路径

4. **回溯判决（Traceback）**：当路径存储深度达到阈值时，回溯最优路径得到判决输出

判决公式（AWGN信道）：
```
max ∑ [- (y_k - μ_k)² / 2σ²]
```

其中μ_k是当前状态对应的发送序列的期望值，σ²是噪声方差。

## 第二章 硬件架构：Viterbi解码器三大功能单元

### 2.1 整体架构

传统Viterbi解码器分为三个基本单元，与MLSE均衡器完全对应：

```
接收软判决信息
      │
      ▼
┌─────────┐
│   BMU   │  Branch Metric Unit（分支度量单元）
│分支度量 │  计算每个时刻各分支度量值
└────┬────┘
      │
      ▼
┌─────────┐
│  ACSU   │  Add-Compare-Select Unit（加-比-选单元）
│  ACS    │  状态度量累加、比较、选择
└────┬────┘
      │
      ▼
┌─────────┐
│   SMU   │  Survivor Memory Unit（幸存路径存储单元）
│路径存储 │  存储幸存路径，回溯判决输出
└─────────┘
```

### 2.2 分支度量单元（BMU）

BMU根据接收的软判决信息计算每个分支的度量值。在硬件实现中：

- **状态均值寄存器**：因为需要计算μ_k的值（状态对应的期望输出），硬件中需要添加状态均值寄存器阵列，连接到BMU。该寄存器存储每个状态对应的发送序列模式。
- **距离计算**：计算|y[k] - μ_k|²，通常使用平方误差电路或查表实现
- **并行计算**：每个时钟周期需计算所有状态的分支度量（2^L × 2条分支），硬件开销巨大

**关键实现挑战**：
- 高速SerDes（56G/112G/224G）要求在极短UI（224G仅8.93ps）内完成度量计算
- PAM4调制下状态数翻倍（4电平导致每个符号携带2bit信息）
- ADC量化精度影响度量准确性——CoreOptics的10.7Gbit/s MLSE接收机使用3bit量化ADC

### 2.3 加-比-选单元（ACSU）

ACSU是Viterbi解码器的计算核心：

**Add（加）**：将分支度量（BM）与前一时刻的状态路径度量（PM）相加，得到候选路径度量

**Compare（比）**：比较到达同一状态的多条候选路径度量

**Select（选）**：选择度量最小的路径作为幸存路径，更新状态寄存器和路径寄存器

```
对每个状态 S_k：
  For each incoming branch from S_{k-1}:
    candidate_PM = PM(S_{k-1}) + BM(transition)
  survivor_PM[S_k] = min(all candidate_PM)
  survivor_path[S_k] = argmin(all candidate_PM)
```

**蝶形运算（Butterfly Operation）**：这是ACSU最重要的优化。对于线性网格图，每对状态共享相同的分支集合，可以合并计算——状态S和状态S'为一对蝶形，它们相互转换且共享度量计算资源。这将ACSU的面积复杂度从O(2^L)降低约一半。

**动态缩放（Dynamic Scaling）**：路径度量随时序累加可能溢出（因为度量值始终为正）。必须实施归一化或动态缩放策略：
- 定时归一化：将所有路径度量减去最小值
- 浮点表示：增加指数位
- 有限状态机的饱和截断

### 2.4 幸存路径存储单元（SMU）

SMU负责存储每状态的幸存路径：

- **路径存储器**：存储每个状态对应的幸存路径（通常用指针或位移寄存器实现）
- **回溯深度（Traceback Depth）**：通常设为状态数的5-10倍，确保路径收敛到全局最优
- **判决输出**：回溯后输出判决比特

**延迟分析**：
- SMU的存储深度直接影响系统延迟
- 回溯深度通常为状态数的5-10倍，例如MLSE-2（8状态）需要40-80个符号延迟
- 对于112G PAM4（波特率53.125GBaud），每符号约18.8ps，80符号延迟约1.5ns

## 第三章 性能优化变体与复杂度降低技术

### 3.1 SMLSE（平均状态均值优化）

**原理**：将同一扩展状态的4个状态均值求平均，用平均后的单一值替代原始4个状态分别计算度量。

例如MLSE-2中，8个状态可以分为2组扩展状态，每组4个状态共用一个平均度量。核心观察是：这4个状态的区别仅在于未被观察的将来符号的影响，在当前时刻的度量计算中，它们的期望值差异不大。

**性能**：大幅降低计算复杂度的同时几乎保持最优性能，是工程落地的首选方案。

### 3.2 MMLSE（最大状态均值优化）

**原理**：在每组扩展状态中，选择度量最大的状态均值（即当前时刻最可能的发送符号组合）。

**性能**：比标准MLSE提升1-2dB，但比SMLSE差。复杂度介于MLSE和SMLSE之间。

### 3.3 两种优化的量化对比

| 指标 | MLSE | MMLSE | SMLSE |
|-----|------|-------|-------|
| 性能（vs理论最优） | 最优 | 差1-2dB | 几乎最优 |
| 计算复杂度 | 高 | 中 | 低 |
| 硬件实现难度 | 高 | 中 | 低 |
| 适用场景 | 理论基准 | 过渡方案 | **工程首选** |

## 第四章 MLSE vs DFE：工程落地的权衡

### 4.1 核心特性对比

| 特性 | MLSE | DFE |
|-----|------|-----|
| 理论基础 | 最大似然序列估计 | 逐符号判决 |
| ISI处理 | 考虑整个序列的全局优化 | 仅反馈已判决符号 |
| 噪声放大 | 无（序列估计） | 无（反馈滤波） |
| 错误传播 | **无** | 有（错误会扩散） |
| 计算复杂度 | 高（指数增长） | 低（线性） |
| 系统延迟 | 长（回溯深度决定） | 短（1符号延迟） |
| 硬件面积 | 大（状态寄存器阵列） | 小（抽头滤波器） |
| 性能 | 最优（AWGN信道） | 次优 |
| 适用场景 | 高性能专用/光通信 | 商用主流SerDes |

### 4.2 为什么商用SerDes主要用DFE而非MLSE

**根本原因：复杂度与性能的性价比**

1. **状态数爆炸**：即使MLSE-1（4状态），ACSU每个状态需要计算2条分支的度量，4状态共8个BM计算单元 + 4个ACS比较器。MLSE-2（8状态）则需要16个BM + 8个ACS，复杂度指数增长。

2. **延迟代价**：MLSE的回溯判决延迟（通常50-100个符号）在高速SerDes中难以接受。以112G PAM4为例，50符号延迟约0.94ns，在链路训练和快速适配场景中成为瓶颈。

3. **信道适配性**：对于典型背板信道（损耗20-30dB@26.6GHz），DFE的2-4个抽头已经足够补偿主要ISI，MLSE的性能增益有限。

4. **功耗**：ACSU+BMU的并行计算带来巨大功耗——一个8状态MLSE的功耗可能是同等DFE的5-10倍。

### 4.3 MLSE的商用先例

尽管存在上述挑战，MLSE在特定场景已有商用落地：

**CoreOptics案例（10Gbit/s光接收机）**：
- 2004年ECOC报道：全球首个集成MLSE功能的光接收机
- ADC规格：25Gsample/s，3bit量化，每比特采样2次
- 调制格式：NRZ
- 性能：可均衡约1600ps/nm的色散（相当于80km标准光纤）
- 2007年推出第二代MLSE技术，应用于城域DWDM平台

**意义**：证明了MLSE在光通信领域的工程可行性。其核心经验是：ADC高速量化 + 专用硬件加速器是实现MLSE的必由之路，而非纯DSP软件实现。

## 第五章 224G SerDes场景下的MLSE演进

### 5.1 224G带来的新挑战

224G PAM4的波特率为106.25GBaud，UI仅8.93ps。MLSE实现面临以下挑战：

**关键路径延迟**：
- MLSE-2（8状态）的ACS运算需要在一个UI内完成，而8.93ps连一个乘法器 latency都cover不住
- 必须采用深度流水线（pipeline）甚至 speculative 执行

**ADC采样率需求**：
- PAM4每个符号需要3个判决阈值
- Nyquist频率达53GHz（106.25GBaud/2）
- ADC ENOB要求6.5-7bit以上（比112G的5-6bit更高）

**状态数爆炸**：
- NRZ下MLSE-1有4状态
- PAM4下每个符号携带2bit，同等MLSE深度对应更多状态（等效状态数4倍）

### 5.2 可能的演进路径

**路径一：ADC-Based架构 + 轻量级MLSE**
- 全ADC采样 + DSP处理
- 在DSP中实现MLSE，而非纯硬件逻辑
- 适合灵活性要求高的场景（可编程链路）

**路径二：Speculative MLSE**
- 将DFE的speculative执行思想移植到MLSE
- 预先计算所有可能的状态转换
- 降低关键路径依赖

**路径三：与神经网络均衡器融合**
- 用DNN替代Viterbi的网格搜索
- 研究显示DNN可以在复杂度可控的前提下逼近MLSE性能
- 是224G乃至下一代均衡的有力候选

### 5.3 MLSE在当前SerDes架构中的定位

当前主流SerDes的均衡链路：
```
TX FFE（预失真）
    → Channel（损耗）
    → CTLE（模拟域线性均衡）
    → ADC/DSP
    → RX FFE（数字域FFE）
    → DFE（判决反馈）
    → CDR（时钟恢复）
```

MLSE在其中的潜在位置：
- **方案A**：替代DFE（作为RX均衡的最后一级）
- **方案B**：与DFE并行（DFE处理短尾ISI，MLSE处理长尾ISI）
- **方案C**：在ADC-Based架构中作为自适应均衡的高级形式

对于224G系统，综合考虑功耗、延迟、复杂度，推荐**方案B**作为过渡——MLSE专注处理长尾色散效应，DFE处理短尾post-cursor ISI，两者协同。

## 第六章 硬件实现关键技术细节

### 6.1 分支度量计算的具体实现

对于二进制调制（PAM2/NRZ），状态μ_k的取值只有两种（对应发送"0"和"1"）。但对于PAM4，状态μ_k的取值有4种，状态均值寄存器的设计复杂度翻倍。

**平方误差电路**：
```verilog
// 简化示意
bm = (received_sample - state_mean) ** 2;
```

**查表实现（LUT）**：
- 将状态均值和接收值编码为地址
- LUT输出预计算的|received - mean|²
- 优点：速度极快，无乘法器delay
- 缺点：面积大（尤其是高精度量化时）

### 6.2 ACSU的流水线设计

由于ACS操作无法在一个短UI内完成，必须引入流水线：

**级联流水线**：
- BM计算 → ACS1 → ACS2 → ... → 回溯
- 每级流水线深度约1-3个时钟周期
- 总延迟 = 流水线级数 × 每级延迟

**折叠架构（Folded）**：
- 共享一套ACS硬件，通过时序复用处理多状态
- 面积小但延迟大，适合功耗敏感场景

### 6.3 功耗优化策略

MLSE均衡器的功耗主要来源：
1. BMU的并行度量计算（最耗能）
2. ACSU的状态比较选择
3. SMU的路径存储读写

**功耗降低技术**：
- **时钟门控**：对不活跃状态进行时钟门控
- **近似计算**：使用分段线性近似替代精确平方
- **动态精度**：根据信道质量动态调整量化精度（好信道用粗量化，差信道用精量化）
- **电源电压调节**：根据时序裕量动态调节VDD

## 第七章 工程落地 checklist

| 检查项 | 要求 | 说明 |
|-------|------|------|
| 状态数选择 | MLSE-1（4状态）或MLSE-2（8状态） | 根据信道ISI长度选择，避免过度设计 |
| 流水线深度 | 覆盖关键路径延迟 | 112G建议3-5级流水线，224G可能需要更深 |
| 动态缩放 | 必须实现 | 防止路径度量溢出，常用定时归一化 |
| 功耗预算 | < SerDes总功耗的15-20% | MLSE不应成为功耗瓶颈 |
| 回溯深度 | 状态数×5~10 | 过浅降低性能，过深增加延迟 |
| 量化精度 | ≥ 4bit（建议6-8bit） | 影响度量准确性，直接关系BER性能 |
| 收敛判定 | 链路训练阶段完成初始收敛 | 运行时的自适应跟踪是难点 |
